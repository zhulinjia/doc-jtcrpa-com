# 等待信号

等待信号节点用于暂停当前工作流，直到接收到指定信号后才继续向下执行。

## 概述

与「[信号触发](../triggers/signal-trigger)」（流程级触发器）不同，等待信号是流程内部的阻塞节点——流程先执行前置操作，中途遇到等待信号时挂起，收到匹配信号后继续。信号可携带数据，存入配置的输出变量供下游引用。

## 使用方法

拖入节点后，配置以下两项：

### 等待信号 ID

要等待的信号标识，与发送方的「[发送信号](../flow-control/send-signal)」节点中的信号 ID 保持一致。信号 ID 完全匹配（区分大小写）才会唤醒流程。支持 `{{变量名}}` 引用。

### 输出变量

接收信号携带数据的变量名。发送方在「发送信号」中填写的「携带数据」JSON，会在信号到达时存入此变量，下游节点通过 `{{变量名.字段}}` 引用。不填则丢弃信号携带的数据。

## 参数参考

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| 等待信号 ID | 文本 | — | 等待的信号标识，与发送方一致。完全匹配才唤醒。支持 `{{变量名}}` 引用 |
| 输出变量 | 文本 | — | 接收信号附带数据的变量名 |

## 常见问题

### 等待信号与信号触发的区别

| | 信号触发 | 等待信号 |
|---|---|---|
| **位置** | 流程开头（触发器） | 流程中间（执行节点） |
| **行为** | 收到信号后启动整个流程 | 阻塞当前路径，收到信号后继续 |
| **适合场景** | 流程被信号驱动启动 | 流程先做前置操作，中途等待外部通知 |

简单来说：流程从头到尾都不需要自己先跑什么 → 用信号触发；流程需要先做一些事再等人 → 用等待信号。
