# 网络拦截

网络拦截触发器监听浏览器网络请求，当匹配到指定 URL 和 HTTP 方法的请求时自动触发工作流。

## 概述

不需要事先知道请求何时发出——网络拦截触发器会持续监听浏览器的 XHR/Fetch 请求。一旦检测到 URL 和请求方法同时匹配的请求，工作流在匹配页面上自动启动。

**注意**：网络拦截在请求**发起时**触发，不是响应返回时。

## 快速上手

**示例**：监听 API 接口请求 → 等待响应返回 → 采集响应数据。

> 此示例为系统默认自带，名称为 **API接口采集【示例】**

<a href="/img/网络拦截示例.png" target="_blank"><img src="/img/网络拦截示例.png" alt="网络拦截示例" style={{maxWidth: 480, width: '100%'}} /></a>

1. 拖入「网络拦截」节点，配置 URL 匹配模式和请求方法
2. 拖入「[等待 HTTP](/components/network-components/wait-http)」节点，等待匹配的请求响应返回
3. 拖入「[数据采集](/components/general-components/data-collection)」节点，从响应数据中提取字段
4. 保存后，当浏览器发起匹配请求时工作流自动启动并采集数据

## 参数参考

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| URL 匹配模式 | 文本 | — | Chrome match patterns 格式，必填。参见 [URL 匹配模式](../../reference/match-patterns) |
| 请求方法 | 下拉选择 | — | 必填。`GET` / `POST` / `PUT` / `DELETE` |

## 常见问题

### 在 Network 面板看到请求但拦截没触发

**现象**：开发者工具 Network 面板中能看到请求，但拦截节点没有触发。

**原因**：URL 匹配模式未命中实际请求路径，或请求方法不匹配，或拦截器在请求发出后才注册。

**解决**：确认 URL 匹配模式和请求方法与 Network 面板中的请求一致；确保拦截节点在流程中的位置早于触发请求的操作。
