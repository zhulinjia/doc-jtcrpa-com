# 元素监控

元素监控触发器监听页面 DOM 变化，子元素增减或属性变化时自动触发工作流。适合需要实时响应页面内容变化的场景。

## 使用示例

**场景一：监控聊天面板，新消息出现时自动采集**

客服系统里，聊天面板不断出现新消息，你想在每条新消息出现时自动记录。

```
元素监控 → 数据采集 → 保存到数据库
```

配置：匹配页面填 `https://chat.example.com/*`，目标元素填 `.message-list`（消息列表容器），监控类型选「子元素增减」。当 `.message-list` 下新增子元素（新消息 `<li>`）时，流程自动启动采集。

**场景二：监控订单列表，状态标签变化时截图**

订单列表页中每个订单的状态标签会从"待处理"变成"已完成"，你想在变化发生时截图存档。

```
元素监控 → 页面截图
```

配置：目标元素填 `.order-status`（状态标签的容器），监控类型选「子元素属性变化」。当状态标签的 class 属性发生变化时自动截图。

## 参数说明

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| 匹配页面 | 文本 | — | Chrome match patterns，限定监控的页面范围，必填。参见 [URL 匹配模式](../../reference/match-patterns) |
| 目标元素 | CSS 选择器 | — | 要监控的父容器元素，必填。支持 [AI 自然语言模式](/core-concepts/css-selectors#ai-自然语言选择器推荐新手使用) |
| 监控类型 | 下拉选择 | — | `childList` — 子元素增减；`attributes` — 子元素属性变化 |

## 常见问题

### 元素变化了但监控没触发

**现象**：页面元素确实在变化，流程没启动。

**原因**：匹配页面 URL 与实际页面不匹配，或目标元素选择器指向了错误的父元素。

**解决**：先确认「匹配页面」是否覆盖当前页；再用 👁 预览按钮验证目标元素是否能命中正确的 DOM 节点。
