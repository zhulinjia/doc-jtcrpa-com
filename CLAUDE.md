# CLAUDE.md

## 项目背景

开始编写或修改文档前，必须先阅读 [Docusaurus](https://docusaurus.io/docs) 官方文档（涉及特定组件时）、项目中的 docusaurus.config.js、sidebars.js 以及目录结构。务必确认文档的分类逻辑、目标读者和国际化（i18n）配置；不要凭经验盲猜路径和侧边栏分类。

## 常用命令

只使用本项目已经声明的命令。优先从 package.json 的 scripts 中确认 Docusaurus 相关的生命周期命令（如本地预览、生产构建）。

不要把示例命令写成项目事实；没有确认过的命令不要写进本文件，也不要留下占位文本。

## 1. 写文档前先想清楚

不要假装已经完全理解产品逻辑，也不要把不确定性藏在模糊的文字里。该说清楚的地方先说清楚。

开始撰写前：

- 明确写出你的假设（例如：假设读者已具备某项前置知识）；不确定就问。
- 如果对功能逻辑或用户操作路径存在多种理解，把几种理解列出来确认，不要悄悄选一个盲写。
- 如果有更简单、更直观的表述方式（或图表表达），要主动说明。
- 发现原始需求或产品逻辑不清晰，立刻停下来指出哪里不清楚，拒绝含糊其辞。
- 如果有引用其它文档知识点，那么需要添加跳转链接。

## 2. 简单优先

只写解决当前信息断层所需的最少文字，不做猜测式扩展，拒绝字数灌水。

- 不添加用户没有要求的页面、章节或 FAQ。
- 不为只出现一次的概念设计复杂的文档结构或多级子目录。
- 不引入未经要求的排版灵活性、自定义 MDX 组件或复杂的 CSS 样式。
- 不为不可能出现的场景堆错误处理。
- 保持大白话，不为了显得专业而堆砌晦涩的名词。如果一句话能说清楚，绝不拆成三句。

自检问题：资深技术作家或产品经理看到这段文档，会不会觉得废话连篇或过度排版？如果会，就删减。

## 3. 外科手术式修改

只碰必须修改的文档段落。只清理自己造成的问题。

编辑已有文档时：

- 不顺手“优化”旁边段落的文字、无关的错别字或中英文空格格式。
- 不重构没有逻辑硬伤的文档结构。
- 严格匹配项目现有的写作风格与语气（包括但不限于：中英文混排加空格规范、专业术语大小写、第二人称指代）。
- 发现无关的废弃文档或死链可以提一句，不要直接删除。

如果你的改动制造了无用内容：

- 删除由本次改动造成的失效链接、未引用的本地图片、无用的 MDX import 导入。
- 不删除本来就存在的死代码/死文档，除非任务明确要求。

检查标准：每一行文字或配置的改动，都应该能追溯到用户这次的明确请求。

## 4. 目标驱动执行

把文档任务转成可验证的目标，然后循环到验证完成。

示例：

- “新增一页指南” -> “先在 sidebars.js 配置好侧边栏路径，写完内容后启动本地服务（pnpm start），验证页面渲染正常且无排版错乱。”
- “更新功能说明” -> “修改完对应的 .md/.mdx 文件后，执行本地构建（pnpm build），确保没有触发 Docusaurus 的 Broken Links（死链）检查错误。”

多步骤任务先写简短计划：

```text
1. [步骤：修改/新增文档] -> 验证：[检查 MDX 语法与文案]
2. [步骤：配置 sidebar/config] -> 验证：[本地 `pnpm start` 查看导航与侧边栏是否符合预期]
3. [步骤：完整构建测试] -> 验证：[运行 `pnpm build` 确保无死链、无编译报错]
```

## 5. 提交

# 提交流程规范 (Commit Workflow)

为了确保代码库的质量、版本可追溯性以及团队协作的顺畅，请在提交至 GitHub 前，严格遵循以下提交流程。

---

## 5.1 自动化质量检查 (Quality Assurance)

在执行 `git commit` 前，请务必执行以下检查，确保代码逻辑无误且风格统一：

- **代码格式化**：运行 `npm run lint` 或 `prettier --write`，移除无效引用，确保符合项目规范。
- **类型检查**：运行 `tsc --noEmit`，确保 TypeScript 类型系统无任何报错。
- **冒烟测试 (Smoke Test)**：在 RPA Playground 或目标浏览器环境下进行快速测试，验证核心交互逻辑（如选择器匹配、等待策略、错误处理）未出现回归。

---

## 5.2 提交说明规范 (Conventional Commits)

请遵循 [Conventional Commits](https://www.conventionalcommits.org/) 规范编写 Commit Message，以便于后续查阅版本演进。

**格式**：`<type>(<scope>): <description>`

### 常用类型说明：

| 类型       | 含义     | 场景示例                                 |
| :--------- | :------- | :--------------------------------------- |
| `feat`     | 新增功能 | 新增一种页面交互方式、新增自动化测试场景 |
| `fix`      | 修复错误 | 修复选择器获取逻辑、修正报错处理边界条件 |
| `refactor` | 代码重构 | 逻辑优化、封装公共组件、性能提升         |
| `docs`     | 文档更新 | 修改 README.md、补充代码内注释           |
| `chore`    | 琐碎任务 | 依赖包升级、构建配置修改                 |

**提交示例**：

- `feat(core): 实现策略模式处理 RPA 页面交互错误`
- `fix(selector): 优化 table 列选择器算法，优先匹配全等`
- `refactor(ui): 调整 TextArea 图标动态布局逻辑`

---

## 5.3 同步与备份 (Synchronization)

完成本地检查与提交描述编写后，请按以下顺序执行操作：

1.  **提交至本地暂存区**：
    ```bash
    git add .
    git commit -m "<type>(<scope>): <description>"
    ```
2.  **更新本地代码**（确保与远程仓库同步）：
    ```bash
    git pull origin master
    ```
3.  **推送至远程 GitHub**：
    ```bash
    git push origin master
    ```
